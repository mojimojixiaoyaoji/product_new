二、需求背景与概述

1. 需求背景

随着企业数据量的快速增长，数据分析已成为企业决策的重要依据。然而，传统的数据分析工具需要专业的技术知识，如SQL编写或复杂BI工具的操作，这使得非技术人员难以直接获取数据洞察。此外，数据分析师的工作效率也受到工具复杂度的限制，无法快速响应业务需求。

智能问数系统（ChatBI）旨在通过自然语言交互，降低数据分析门槛，让更多用户能够轻松获取数据洞察，同时提高数据分析师的工作效率。

2. 产品目标

实现通过自然语言提问快速获取数据结果、图表和分析报告的功能，简化数据分析流程，让用户无需编写代码即可快速获取数据洞察。

3. 产品价值

提高数据分析效率，降低数据分析门槛，促进数据驱动决策，为企业创造更大的业务价值。

三、产品概述

智能问数系统（ChatBI）是一个基于自然语言交互的数据查询和分析工具，用户可以通过自然语言提问获取数据结果、图表和分析报告。

1.1 需求场景描述

场景一：数据分析师日常工作

背景：数据分析师需要定期分析业务数据，生成报表和图表。

问题：传统的数据分析工具需要编写SQL或使用复杂的BI工具，操作繁琐，效率低下。

解决方案：使用智能问数系统，通过自然语言提问快速获取数据结果和图表。

场景二：业务管理者决策支持

背景：业务管理者需要基于数据做出决策，但缺乏专业的数据分析技能。

问题：无法快速获取所需的数据洞察，依赖于数据分析师的支持。

解决方案：使用智能问数系统，通过简单的自然语言提问获取决策所需的数据和报告。

场景三：跨部门数据协作

背景：不同部门的人员需要共享和分析数据，促进协作。

问题：数据共享和理解存在障碍，沟通成本高。

解决方案：使用智能问数系统，通过统一的自然语言接口获取和分享数据洞察。

五、用户故事+ GWT定义

1. 用户故事规范

格式要求：作为 [角色]，我希望 [实现的功能/目标]，以便 [业务价值/用户收益]。

核心原则：每个用户故事对应1-2个核心场景，配套GWT定义，明确场景边界与预期结果，为研发、测试提供清晰依据。

2. 用户故事+GWT定义示例（按角色分类）

（1）数据分析师

用户故事：作为数据分析师，我希望通过自然语言快速查询数据并生成图表，以便高效完成数据分析任务。

GWT定义-场景1：自然语言查询
  • Given：用户已登录系统，角色为数据分析师，拥有数据分析权限；系统正常运行，已对接数据源；用户明确查询需求。
  • When：用户在输入框中手动输入「查询最近一周的水泥强度预测情况」，点击查询按钮。
  • Then：系统在3秒内返回查询结果；显示数据总结、详细数据表格和趋势图表；生成3条相关的下一步建议；无异常提示。

（2）业务管理者

用户故事：作为业务管理者，我希望通过简单的自然语言提问获取数据洞察和分析报告，以便做出数据驱动的决策。

GWT定义-场景：智能报告生成
  • Given：用户已登录系统，角色为业务管理者，拥有报告生成权限；系统已存储最近一次查询结果。
  • When：用户在输入框中输入「查询报告」，点击发送按钮。
  • Then：系统在5秒内完成报告生成并显示完整的分析报告；报告包含数据摘要、趋势分析、洞察建议等内容；支持导出为DOC格式。

（3）生产技术人员

用户故事：作为生产技术人员，我希望通过语音输入快速查询生产数据，以便在现场操作时高效获取信息。

GWT定义-场景：语音输入
  • Given：用户已登录系统，角色为生产技术人员，拥有数据查询权限；系统语音识别功能正常，麦克风权限已开启。
  • When：用户点击语音输入按钮，朗读「查询最近一周的水泥强度预测情况」，完成后点击确认识别。
  • Then：语音识别准确率≥90%，正确转换为文本查询条件；系统在3秒内返回对应数据、趋势图表及总结；识别失败时，返回提示：「语音识别失败，请重新朗读」。

六、数据权限

明确不同角色对数据的访问、操作范围。

  • 默认：用户仅可见有调用权限的数据。
  • 数据访问：根据用户角色和权限级别，限制对敏感数据的访问。
  • 操作权限：不同角色拥有不同的操作权限，如数据导出、报告生成等。

七、验收标准

列出功能上线前必须满足的测试与验证项，通常关联AC（Acceptance Criteria）。

功能验收标准

  • AC-001：支持通过自然语言提问获取数据结果，响应时间不超过3秒
  • AC-002：支持线图、柱图、饼图三种类型，图表渲染时间不超过2秒
  • AC-003：支持基于最近查询结果生成分析报告，生成时间不超过5秒
  • AC-004：支持新建、切换、删除会话，会话数据正确保存
  • AC-005：支持导出表格为CSV、图表为PNG、报告为DOC
  • AC-006：支持左侧会话列表宽度拖拽调节，范围260px-600px

性能验收标准

  • AC-007：系统响应时间不超过3秒
  • AC-008：图表渲染时间不超过2秒
  • AC-009：报告生成时间不超过5秒
  • AC-010：系统连续运行24小时无崩溃

用户体验验收标准

  • AC-011：符合现代设计风格，布局合理，色彩协调
  • AC-012：拖拽调节无卡顿，按钮点击有反馈
  • AC-013：适配不同屏幕尺寸
  • AC-014：支持Chrome、Firefox、Safari、Edge主流浏览器

八、核心业务逻辑与功能流程

1. 功能清单

功能名称 核心业务逻辑 异常业务逻辑和注意事项
自然语言查询 用户通过自然语言输入查询需求，系统分析意图并返回数据结果 查询无结果时显示友好提示，提供示例查询建议
图表生成与切换 根据查询结果生成默认线图，支持切换为柱图、饼图 数据为空时不生成图表，显示数据为空提示
智能报告生成 基于最近一次查询结果生成分析报告 无历史查询数据时提示用户先查询具体数据
会话管理 支持新建、切换、删除会话，会话数据本地存储 删除会话时弹出确认提示，防止误操作
语音输入 通过Web Speech API实现语音识别，转换为文本查询 语音识别失败时显示错误提示，引导用户重新输入
界面调节 支持左侧会话列表宽度拖拽调节，范围260px-600px 拖拽时限制最小和最大宽度，防止超出范围

2. 业务流程

用流程图或文字步骤描述核心业务的完整流转路径。

核心业务流程

  1. 用户打开智能问数系统，系统初始化并加载历史会话
  2. 用户通过文本或语音输入查询需求
  3. 系统分析查询意图，处理数据
  4. 系统生成数据结果、图表和下一步建议
  5. 用户查看结果，可进行图表切换、数据导出等操作
  6. 用户可输入"查询报告"生成分析报告
  7. 用户可管理会话（新建、切换、删除）

十、交互流程

10.1 用户操作流程

10.1.1 自然语言查询流程

  1. 初始化：用户打开应用，系统显示默认提示词和示例问题
  2. 输入查询：用户通过文本输入、语音输入或点击示例问题提交查询
  3. 加载状态：系统显示"正在查询数据，请稍候..."的加载提示
  4. 结果展示：系统生成查询结果卡片，包含数据表格、图表等
  5. 后续操作：用户可以查看结果、导出数据、切换图表类型或点击下一步建议

10.1.2 会话管理流程

  1. 新建会话：用户点击「+ 新对话」按钮创建新会话
  2. 自动开场白：系统自动展示智能问数助手的开场白
  3. 切换会话：用户在历史会话列表中点击会话项切换会话
  4. 删除会话：用户点击会话项旁边的删除按钮删除会话

10.1.3 报告生成流程

  1. 数据查询：用户先进行至少一次数据查询
  2. 请求报告：用户输入"查询报告"并提交
  3. 加载状态：系统显示"正在基于最近一次查询结果生成报告..."的加载提示
  4. 报告展示：系统生成并显示完整的分析报告
  5. 导出报告：用户可以点击导出按钮下载报告为DOC格式

10.2 系统响应流程

10.2.1 查询处理流程

  1. 接收请求：系统接收用户的查询请求
  2. 意图分析：系统分析查询意图，确定查询类型
  3. 数据处理：系统根据查询类型处理数据
  4. 结果生成：系统生成数据表格、图表等结果
  5. 建议生成：系统根据查询类型生成下一步建议
  6. 结果返回：系统将结果返回给用户

十一、技术实现

11.1 技术栈

11.1.1 前端技术

  • 基础技术：HTML5 + CSS3 + JavaScript
  • 图表库：ECharts 5.4.3 - 用于生成交互式图表
  • 存储：localStorage - 用于会话数据本地存储
  • 语音识别：Web Speech API - 用于实现语音输入功能

11.2 核心功能实现方案

11.2.1 会话管理实现

  • 数据存储：使用localStorage存储会话数据，确保页面刷新后数据不丢失
  • 会话结构：包含会话ID、会话名称、创建时间、消息记录等信息
  • 会话操作：实现新建、切换、删除会话的核心逻辑

11.2.2 图表功能实现

  • 图表初始化：使用ECharts.init()初始化图表实例
  • 默认图表：所有图表默认展示为线图
  • 类型切换：实现线图、柱图、饼图之间的切换逻辑
  • 图表配置：根据不同数据类型生成相应的图表配置
  • 响应式：监听窗口 resize 事件，自动调整图表大小

11.2.3 界面调节实现

  • 拖拽逻辑：实现鼠标拖拽分隔线调整宽度的功能
  • 宽度限制：设置最小宽度260px，最大宽度600px
  • 视觉反馈：拖拽时显示调整光标，提供视觉反馈
  • 平滑过渡：使用CSS transition实现宽度调整的平滑过渡效果

11.2.4 报告生成实现

  • 触发机制：监听用户输入"查询报告"关键词
  • 数据来源：基于最近一次数据查询结果生成报告
  • 报告结构：包含核心结论、数据表现、业务建议等部分
  • 导出功能：使用Blob对象和a标签实现报告导出为DOC格式

11.2.5 语音输入实现

  • API使用：使用浏览器Web Speech API实现语音识别
  • 状态管理：实现语音识别的开始、停止、错误处理等状态管理
  • 自动提交：语音识别完成后自动填充输入框并提交查询

十二、核心业务逻辑及功能闭环

12.1 核心业务流程

12.1.1 系统初始化流程

  1. 页面加载：用户打开智能问数系统页面
  2. 数据初始化：系统初始化会话管理模块，加载本地存储的历史会话
  3. 界面渲染：系统渲染左侧会话列表和右侧默认提示区域
  4. 状态设置：系统设置初始状态，准备接收用户输入

12.1.2 查询处理流程

  1. 接收输入：系统接收用户的文本输入、语音输入或示例问题点击
  2. 意图识别：系统分析查询意图，确定查询类型（水泥强度、项目工时等）
  3. 数据处理：系统根据查询类型处理相应的数据
  4. 结果生成：系统生成数据表格、图表等结果内容
  5. 建议生成：系统根据查询类型生成相关的下一步建议
  6. 结果展示：系统在右侧结果区展示查询结果和建议

12.2 功能点闭环

12.2.1 数据查询闭环

  • 输入：用户输入自然语言查询
  • 处理：系统分析查询意图，生成查询结果
  • 展示：系统显示数据表格和图表
  • 建议：系统提供下一步查询建议
  • 交互：用户可以点击建议继续查询，形成闭环

12.2.2 图表功能闭环

  • 初始化：系统生成默认线图
  • 切换：用户可以切换到柱图或饼图
  • 更新：系统实时更新图表类型和样式
  • 导出：用户可以导出图表为PNG格式
  • 反馈：系统提供图表类型切换的视觉反馈

12.2.3 报告生成闭环

  • 数据准备：用户先进行至少一次数据查询
  • 触发：用户输入"查询报告"关键词
  • 生成：系统基于最近查询结果生成分析报告
  • 展示：系统显示完整的分析报告
  • 导出：用户可以导出报告为DOC格式

12.2.4 会话管理闭环

  • 创建：用户创建新会话
  • 保存：系统自动保存会话数据到本地存储
